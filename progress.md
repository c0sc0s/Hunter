# Hunter Progress

Last updated: 2026-06-04

## Current Facts

- Hunter uses a source-first capture architecture.
- Capture is browser-snapshot only: `POST /api/items` requires a snapshot from the browser extension and rejects requests without one with `400`.
- The server no longer performs any URL fetch, oEmbed fetch, or PDF download; source adapters recognize content from the snapshot the extension delivers.
- Source adapters cover generic web pages, X posts, Feishu/Lark pages, and PDFs (snapshot text). Video hosts (YouTube, Vimeo, Bilibili, etc.) are recognized through `genericWebAdapter` via JSON-LD `VideoObject` + og:\* extraction, with no host-specific adapter — the same pipeline serves any standards-compliant video page.
- Generic Web recognition consults a pure Content Form Detector (og:type, Twitter card, JSON-LD @type, oEmbed discovery link, `<video>` count) and promotes `sourceType` to `video` for video-shaped pages on unknown hosts, while host-routed adapters stay authoritative.
- Generic Web extraction reads `VideoObject` / `AudioObject` JSON-LD nodes from the snapshot when the form detector identifies the page as video or audio, surfacing the structured uploader name, upload date, and thumbnail ahead of Defuddle's site-level author and the generic og:image fallback. ISO 8601 duration parsing is available for a future `LibraryItem.duration` slice.
- Cover image URLs are upgraded from `http://` to `https://` when the captured page is https, mirroring the browser's mixed-content policy. This fixes the case where B站 VideoObject.thumbnailUrl (and other CDN-fronted sites) emits http URLs that the https Hunter web client would silently refuse to render.
- Cover image selection filters known platform-default preview images, including X's logged-in default `abs.twimg.com/rweb/ssr/default/.../og/image.png`, so those placeholders cannot beat real article or post media. The X adapter now prefers attached tweet media from `pbs.twimg.com/media` regions over generic meta images; re-running the stored snapshot for `https://x.com/idoubicc/status/2062152804014436508` returns `https://pbs.twimg.com/media/HJ4WUQpaMAEcEkw?format=jpg&name=medium` instead of X's default og image.
- The extension extractor lifts `<script type="application/ld+json">` scripts from anywhere in the document (not just `<head>`) into the serialized snapshot's `<head>` so they survive the 180KB cap. Without this, B站 (which injects VideoObject ld+json near the bottom of `<body>`) silently lost the JSON-LD before recognition ran.
- The extension extractor ranks image candidates before applying the fixed candidate cap. It collects metadata images, focused-root images, `srcset`/lazy-loading attributes, and inline `background-image` URLs, with bonuses for source-specific media regions like X `tweetPhoto`. This keeps real article/post images from being displaced by avatars, emoji, icons, default platform previews, or sidebar images. The popup preview uses the same candidate intent for article images and X media ranking.
- The web client renders `Cover` images through a real `<img referrerPolicy="no-referrer">` element instead of CSS `background-image`. CSS background-image cannot opt out of the Referer header, so hotlink-protected CDNs (B站 `i*.hdslb.com` returns 403 on cross-origin Referer) would silently refuse to serve a stored cover URL even though the API delivered the correct https link. The `<img>` element opts out of Referer, the CDN returns 200, and an `onError` fallback restores the Sparkles placeholder if the image fails for any other reason.
- Cover URLs from B站 CDN (`i*.hdslb.com`) carry an `@<width>w_<height>h.<ext>` resize directive that defaults to a tiny SEO thumbnail (e.g. og:image `@100w_100h_1c.png` ~3KB, VideoObject thumbnailUrl `@189w_107h.webp` ~6KB). The CDN accepts arbitrary `@<W>w.<ext>` directives, so `shared/coverImageUrl.ts#upgradeCdnCoverResolution` rewrites any hdslb resize directive smaller than 1280w to a canonical `@1280w.webp` (~104KB) that looks crisp on retina list cards and detail panels. The helper runs both at capture-time inside `selectCoverImage` (so new items store the upgraded URL) and at render-time inside the web client `Cover` component (so legacy items captured before this rewrite landed still display crisp covers without a database migration). Cover URLs that already requested ≥1280w are left untouched, and non-hdslb hosts are ignored to avoid touching CDNs that don't follow this convention.
- For video pages, `contentSignals.ts#pickMediaSummary` overrides `buildSummary` and uses `extracted.excerpt` directly. Without this, `ItemCard` (which reads only `item.summary`) would render whatever `buildSummary` distilled from `contentHtml` segments — i.e. page chrome like recommendation rails, comment text, and sidebar paragraphs — even after the adapter had already extracted the uploader's description into `excerpt`. The detail panel's `summary || excerpt || readableText` fallback chain would never trigger because `summary` was always non-empty for video pages. Audio pages will plug in the same way once `SourceType` gains the literal.
- For video/audio pages, the structured `VideoObject` / `AudioObject` `description` from JSON-LD beats `og:description` and snapshot body text both in the item's `description` field and in the `excerpt` shown on the library card. Hosts like B站 ship no meta description at all (so today snapshot.excerpt fell back to random page body text) AND emit the JSON-LD description as a percent-encoded string. `jsonLd.ts#safeDecodeIfUriEncoded` handles the decoding with a conservative heuristic — only strings that contain `%XX` pairs AND zero whitespace get decoded (because natural text always contains literal spaces while encoded text encodes spaces as `%20`), with malformed sequences falling back to the original. The new `JsonLdMetadata.formDescription` field is gated on form="video"/"audio" so plain articles continue to prefer the page's own og:description / meta description. YouTube emits an empty `VideoObject.description` (description is JS-rendered), so `jsonLd.ts#asText` collapses empty/whitespace strings to `undefined` and the genericWeb `??` chain falls through to `og:description` — the same removal of the bespoke `videoAdapter` lets every video host benefit from this pipeline instead of being shorted to a title-only adapter.
- The extension popup previews the same cover the library card will show. `extension/src/coverPreview.js` exports `collectCoverCandidatesInPage` (injected into the captured tab via `chrome.scripting.executeScript` to read og:image / Twitter card / JSON-LD VideoObject.thumbnailUrl) and `upgradeCdnCoverResolution` (a hand-synced copy of the shared TS helper, since popup.js cannot import the TypeScript module directly). The popup renders the candidate through `<img referrerPolicy="no-referrer">` with `object-fit: cover`, applies the hdslb resize rewrite, and falls back to the Hunter brand mark when no candidate exists. The earlier favicon-as-background path is gone — stretching a 16×16 favicon to 360px read as a render bug, and the brand mark is a cleaner empty state. Covered by `pnpm test:cover-preview` (jsdom fixtures for both helpers).
- The extension popup is now visually aligned with the main Hunter client: neutral shadcn-style dark tokens, compact card/input/button treatments, source and snapshot badges, a chevron API details row, status-dot feedback, and a real icon+text Save button whose accessible name remains `Save`. A local browser preview measured a 376px popup width, about 584px content height, and no horizontal overflow.
- The extension now gates save entry points to supported article, video, and X post resources. `extension/src/contentSupport.js` runs a lightweight deterministic preflight inside the captured tab: source-specific X route checks allow post detail and X article URLs while rejecting feeds/timelines before generic DOM scoring; non-X pages use Open Graph type, Twitter player cards, JSON-LD `@type`, focused article roots, weak `<video>` tiebreakers, and server-aligned YouTube/Vimeo host hints. Toolbar popup opens as usual, but only supported pages reveal the add form; unsupported pages render an unsupported-type bubble. Background and context-menu saves reuse the same gate and show a notification instead of posting unsupported snapshots. The gate is UX-only: final `sourceType` and extraction quality still come from server recognition.
- Extension API-base resolution now probes a user-configured off-list localhost base before the default 4317–4319 candidates. This prevents installed-extension tests or custom local API ports from accidentally saving into another running Hunter instance on 4317.
- Repository behavior is abstracted behind JSON and SQLite adapters.
- The SQLite schema migration drops `connectors`, `connector_credentials`, `required_connector`, `capture_method`, and `source_access`, and remaps any historical `enrichment_state='needs_connector'` rows to `failed`. The JSON store strips the same legacy fields on load.
- `EnrichmentState` is `processing | ready | partial | failed`; `needs_connector` is gone.
- Recognition is deterministic and local; it must not depend on AI models.
- The web client uses manual Reload; it does not poll for background recognition.
- The web sidebar shows recent Capture Events with a manual Reload control and `/events` command; the Connector panel was removed when capture became snapshot-only.
- The extension caps browser snapshot payloads before posting to the local API. The "Save link" right-click menu was removed; the toolbar action popup, "Save page" right-click, and background save path are the remaining entry points and all carry a snapshot.
- Popup Save delegates to the extension background save-tab pipeline instead of owning duplicate extraction and POST logic.
- Generic web recognition has a pure quality gate for candidate selection, extraction state, confidence, selected-text parser skip, and lazy Readability fallback.
- Parser HTML is sanitized with DOMPurify before storage as Sanitized Content HTML.
- Saved Items carry Recognition Version, recognized timestamp, recognition duration, phase timing, and Content Hash for parser upgrades and reprocessing comparisons.
- Saved Items and recognition jobs use bounded Capture Input for refresh/reprocessing while public API responses strip it to avoid returning large snapshots.
- Capture Events record queued captures, recognition outcomes, manual refresh outcomes, snapshot byte sizes, timing, content hashes, and errors without storing raw snapshot bodies.
- Browser golden journey runs through `pnpm golden:browser` and is included in `pnpm verify`.
- Installed Chrome extension golden runs through `pnpm golden:extension` and is included in `pnpm verify`. It covers background save messaging, unsupported-resource rejection, and visible popup Save; it attempts `chrome.action.openPopup()` toolbar smoke and skips only Chromium's environment-specific `active browser window` automation limitation.
- Visual golden runs through `pnpm golden:visual` (with `pnpm golden:visual:update` to refresh baselines) and is included in `pnpm verify`. It seeds a deterministic ready snapshot item instead of any connector-needed item.
- The web library UI is on the refreshed shadcn-based dark theme with neutral macOS-style scrollbars. Desktop detail is a fixed right-column panel; the first visible item is selected by default after load/filter changes, and the panel shows a refined empty state when the current view has no item to show. The middle library uses a masonry column layout so cards can keep natural, content-driven heights. Items without real cover images do not render placeholder covers: middle cards switch to a compact text-first layout, and the right detail panel switches to a matching title information block. Fast local loading uses a delayed skeleton reveal when current content can stay visible and a minimum visible duration once skeleton appears, avoiding one-frame flashes. The columns, cards, and no-cover detail header share a subtle top gradient surface treatment for extra depth while keeping scrollbars neutral macOS-style. Desktop intentionally has no visible app header, and the left sidebar no longer renders the Hunter logo/title row. A transparent left-top window control slot reserves space for Electron's hidden macOS titlebar traffic lights and owns the sidebar collapse/expand button; sidebar navigation starts below it when expanded, and middle content shifts below it when collapsed. The desktop sidebar collapse is persisted and removes the sidebar column entirely (`0px`) instead of showing a mini menu; the left-top control restores the full 292px sidebar. The middle library pane owns desktop scrolling. Mobile keeps the Sheet detail path and does not open it until item selection.
- The desktop shell now uses Electron instead of the previous Rust shell. `pnpm dev` runs `electron/dev.mjs`, starts Vite on strict port 5173, opens an Electron window, and lets `electron/main.cjs` supervise the Node API sidecar. The sidecar still announces `HUNTER_API_PORT=<port>` on stdout and binds the first available port in `4317-4319`. Electron preload exposes only `window.hunterDesktop` methods for API-base discovery, api-ready events, and login autostart. The renderer no longer imports legacy desktop APIs. Production Vite output uses relative asset URLs so packaged Electron `file://` rendering loads JS, CSS, and brand assets.
- `pnpm test` covers URL normalization, recognition metadata, recognition timing, Capture Events, capture input budget, content quality, content signals, content HTML sanitization, cover image scoring, Extracted Content contract, Content Form Detector, recognition fixtures, source adapter contracts, the snapshot-required API guard, focused extension snapshots, and SQLite repository behavior.
- ESLint runs with `--max-warnings=0`; Prettier is enforced through `pnpm format:check`; GitHub Actions runs the full `pnpm verify` gate for pull requests and pushes to `main`.

## Open Gaps

- `linux-x64` visual baselines are not committed yet; GitHub Actions temporarily allows missing platform baselines until they are generated from a CI-compatible Linux environment.
- Snapshot-driven block fidelity for Feishu/Notion-style sources and snapshot-derived transcripts for video/audio pages are still open product directions.
- Image-heavy PDFs still need an OCR adapter to leave `partial` or `failed`.

## Next Step

The capture pipeline is intentionally narrow now: only browser-snapshot input, only snapshot-driven adapters, no connector code path. The next useful slice is either: (1) release-engineering for Electron signing/notarization/auto-update/data-dir migration, (2) extend `SourceType` and `LibraryItem` to carry video duration so the parsed seconds render as a duration badge in the library, (3) wire the audio Content Form into the host-routed video adapter / `SourceType` so podcast pages are recognized as their own type, (4) snapshot-quality improvements per source (richer Feishu block recognition, snapshot-based video transcript capture, OCR for image PDFs), or (5) generate `linux-x64` visual baselines.
