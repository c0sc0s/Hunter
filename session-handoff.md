## Handoff Notes

- Capture is browser-snapshot only. `POST /api/items` requires a snapshot and returns `400` without one. Do not add server-side URL or oEmbed fetch back into source adapters.
- Refresh requires the stored snapshot; items without a snapshot cannot be refreshed.
- The browser extension entry points are the toolbar popup, the "Save page" right-click, and the background save-tab pipeline; all three carry a snapshot. The "Save link" right-click menu was removed.
- The toolbar popup no longer gates the add form by resource type. `extension/src/contentSupport.js` still contains the earlier article/video/X detector, but `CONTENT_SUPPORT_GATE_ENABLED=false` disables it while the algorithm is redesigned. Toolbar, background/runtime, and context-menu saves now post snapshots for any injectable browser page; do not send plugin-guessed `sourceType` to override server recognition.
- The static extension popup is intentionally aligned with the main web client's dark UI: neutral primary button, compact cards/inputs, source and snapshot badges, API details chevron, status-dot feedback, and Hunter icon assets. Keep `#tags`, `#note`, `#apiBase`, `#saveButton`, and the success text `Saved. Click Reload in Hunter.` stable for the installed extension journey.
- `extension/src/apiBase.js#resolveApiBase` validates `/api/health` as `service="hunter-api"` and, in automatic/default mode, ranks healthy local sidecars by `owner` (`electron-packaged` > `electron-dev` > `standalone`) and `startedAt` before choosing a 4317–4319 API. This prevents extension saves from landing in a stale dev/test data directory when the visible Electron app is using another port. Non-default user-configured API bases remain manual overrides.
- `EnrichmentState` is `processing | ready | partial | failed`; the SQLite migration remaps legacy `needs_connector` rows to `failed` and drops the `connectors`, `connector_credentials`, `required_connector`, `capture_method`, and `source_access` schema objects. The JSON store strips the same legacy fields on load.
- Source adapters work only with snapshot data: Feishu, X, PDF, video, and generic web all read from `snapshot.html`, `snapshot.textContent`, or `snapshot.selectedText`.
- Agent classification is optional post-recognition work, not content recognition. The sidebar Settings modal configures Ollama, DeepSeek, or another OpenAI-compatible provider through `GET /api/agent/settings` and `PATCH /api/agent/settings`; raw API keys stay server-side and the renderer only sees `apiKeyConfigured`. `GET /api/agent/local-llm` reports the configured provider/model, `POST /api/agent/items/:id/classify` classifies one item, and `POST /api/agent/items/classify-missing` incrementally classifies ready/partial items with missing or stale `contentCategory` by `contentHash`. Each item is one LLM call, even when triggered through the batch endpoint. Before each call, Hunter passes existing generated categories to the model; returned categories are normalized by `existingCategoryId` or label match so later items reuse prior categories. `LibraryStats.agentCategories` powers the Search-below AI topic chips, and `GET /api/items?agentCategoryId=<id>` filters by generated category. LLM output must not decide `sourceType`, extraction state, recognition confidence, deterministic tags, user tags, or Content Signals. Env knobs remain fallback/defaults: `HUNTER_LLM_PROVIDER=ollama|deepseek|openai-compatible`, `HUNTER_LOCAL_LLM_BASE_URL`, `HUNTER_LOCAL_LLM_MODEL`, `HUNTER_LOCAL_LLM_TIMEOUT_MS`, `HUNTER_LOCAL_LLM_TEMPERATURE`, `DEEPSEEK_API_KEY`/`HUNTER_DEEPSEEK_API_KEY`, `HUNTER_DEEPSEEK_MODEL`, `HUNTER_DEEPSEEK_BASE_URL`, `HUNTER_OPENAI_COMPATIBLE_BASE_URL`, `HUNTER_OPENAI_COMPATIBLE_API_KEY`, and `HUNTER_OPENAI_COMPATIBLE_MODEL`.
- Cover recognition stays snapshot-only. X's default logged-in `abs.twimg.com/rweb/ssr/default/.../og/image.png` is treated as a weak platform placeholder, and X attached media from `pbs.twimg.com/media` is preferred. The extension extractor ranks structured image candidates from metadata, focused-root images, `srcset`/lazy attributes, and inline `background-image` URLs before the fixed candidate cap; the server then re-scores preferred JSON-LD thumbnails, HTML metadata/images, and those structured extension candidates together so strong in-content images can beat generic site-level Open Graph images. If every candidate is weak or unusable, items simply have no cover.
- `pnpm test` covers URL normalization, recognition metadata and timing, Capture Events, capture input budget, content quality, content signals, agent local LLM behavior, content HTML sanitization, cover image scoring, Extracted Content contract, Content Form Detector, JSON-LD form-aware picker and ISO 8601 duration parsing, recognition fixtures (including VideoObject author/uploadDate/thumbnail surfacing), source adapter contracts, the snapshot-required API guard, focused extension snapshots, and SQLite repository behavior. Connector-secret and html-fetch tests were removed with the rest of the connector path.
- Agent classification is available from the detail panel through the Agent section's Classify/Reclassify action. Results persist in both JSON and SQLite repository adapters.
- `pnpm golden:browser`, `pnpm golden:extension`, and `pnpm golden:visual` were updated to seed snapshot-only items and no longer assert connector UI; `pnpm golden:visual:update` may need to be re-run on a clean checkout if the seeded ready item shifts layout. The earlier `pnpm golden:extension` empty-list failure was caused by localhost probing preferring a live 4317 dev server over the golden's configured random API port; `resolveApiBase` now respects configured non-default local bases and auto-ranks default 4317–4319 sidecars by health owner. The installed-extension golden writes the random API base before the first save and waits for `/api/health` from the extension context. It attempts `chrome.action.openPopup()` but skips only Chromium's `active browser window` automation limitation; the visible popup Save path still runs.
- Packaged Electron rendering depends on `vite.config.ts` setting `base: "./"`. Without it, `release/mac-arm64/Hunter.app` loads `dist/index.html` over `file://` but JS/CSS resolve to `/assets/...`, producing a blank window. The packaged smoke should assert `window.hunterDesktop`, `getApiBase()`, no renderer `window.require`, visible `Library` text, and no `ERR_FILE_NOT_FOUND` console messages.
- The library UI detail behavior is intentional: desktop detail is a fixed right-column panel, the first visible item is selected by default, and the panel shows an empty state when the current view has no item to show. The middle library uses CSS columns for a masonry layout, so cards use natural content-driven heights instead of a uniform grid row height. Items without real cover images do not render placeholder covers; cards use a compact text-first layout and detail uses a matching title information block. Fast local loading avoids skeleton flicker with a delayed reveal for existing-content refreshes and a minimum visible duration once skeleton is shown. The columns, cards, and no-cover detail header share a subtle top-gradient surface treatment; scrollbar styling stays neutral macOS-like rather than theme-colored. Desktop intentionally has no visible app header, and the left sidebar no longer renders the Hunter logo/title row. A transparent left-top window control slot reserves space for Electron's hidden macOS titlebar traffic lights and owns the sidebar collapse/expand button. Desktop sidebar collapse is intentional and persisted in localStorage: expanded width is 292px, collapsed width is 0px with no mini menu, and the left-top control restores the full sidebar. Mobile detail remains a Sheet and is isolated from the desktop panel by a media query.
- `.github/workflows/verify.yml` still runs `pnpm verify` on pull requests and pushes to `main` with Node 22, pnpm 10.33.0, Playwright Chromium, and Xvfb. GitHub Actions keeps `HUNTER_VISUAL_ALLOW_MISSING_BASELINE=true` until `linux-x64` baselines are generated from a Linux environment.
